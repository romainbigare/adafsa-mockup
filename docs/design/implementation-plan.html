Wafra dashboard — implementation plan

How to turn the mockup into Proposal B, A, and A2 on top of the existing buildless codebase — sharing one foundation so the path B → A → A2 never throws work away.

Stack: vanilla JS · window.Wafra namespace · classic <script>s · Leaflet · no build step. All estimates are relative T-shirt sizes for one developer — calibrate to your own velocity.

Start here — the six modules don't match today's code

Before any layout work: the contract has six modules, but the code only models three of them as first-class "modules" (js/dashboard/modules.js → Irrigation / Yield / Water, as boundary-colouring bands). The other three live as taxonomy browsers (land-use / crops / trees datasets) driven by viewportStats.js + categoryTables.js. Every proposal needs these unified first.

Shared foundation build once

Four pieces every proposal draws on. This is the bulk of the real work; the layouts on top are comparatively thin.

F1Module registry — the backbone

One descriptor per contract module, so nav (A/A2) and the dial (B) read from the same source of truth. New file js/dashboard/moduleRegistry.js: an array of 6 entries, each carrying { key, label, icon, feePct, kind, dataset, colourFn, kpis(), legend, tableColumns }.

  • Irrigation / Yield / Water: wrap the existing MODULES + bands already in modules.js.
  • Palms & Fruit Trees, Crop Monitoring, Structures: new descriptors reading the trees / crops / land-use taxonomies.
  • New mock metrics in js/mock/metrics.js (stay behind the mock boundary → swap for API later): per-farm tree count, canopy-health index, top cultivar (palms); cultivated / fallow area (crop monitoring); structure tier (structures).
  • Test: test/moduleRegistry.test.js — every module returns valid KPIs + a colour for a sample farm.

Needed by A · A2 · B. This is what makes "six modules" real instead of a mix of two mental models.

F2Aggregation + render at 25k farms

Today plotsLayer.streamFeatures() inserts markers one-at-a-time (~45s for 5.3k; 25k would be minutes), and cluster colour uses majority value — which hides the critical farms a module exists to surface.

  • Bulk-insert with clusterGroup.addLayers(array) instead of per-feature addLayer.
  • Precompute per-module region rollups once (the scorecard numbers) so cards never touch the map.
  • Polygons only at zoom ≥ threshold; clusters below show count (reserve colour for the value at farm zoom, or split-tint the cluster).
  • Switch polygon rendering to L.canvas(); plan vector tiles for the real 25k dataset later.

Needed by A · A2 · B. A2's home map is the most demanding consumer — it draws every farm at once.

F3Scorecard card component

New js/dashboard/scorecard.js renders a module card from F1: headline number, status chip (good / warning), and a small inline-SVG micro-chart (band strip or sparkline). One component, two sizes (A's big card, A2's mini tile).

Needed by A · A2. Not B.

F4Ranked "attention list"

A per-module farm table sorted by that module's severity (most-stressed first), searchable by farm / owner, paginated — replacing the idea of dumping 25k rows. Built on the existing js/dashboard/dataTable.js factory and the farmTable.js pattern.

Needed by A · A2 (module pages) · B (its "Farms ranked" tab).

BOne map, one dial small–med · milestone 1

Ship first: lowest risk, mostly deletion, demo-ready. Keeps the single-page GIS workspace but collapses three control surfaces into one.

Changes

  • index.html — replace the seven stacked .cat-panel / .mod-panel accordions and the eight .lb-tabs with one segmented VIEW selector (7 options from F1) + a single legend panel. Bottom sheet → 2 tabs (Summary · Farms ranked). News → a bell button.
  • New js/dashboard/viewSelector.js — select(moduleKey) sets the active module, recolours the map via the F1 colourFn (reusing plotsLayer.applyColoring), renders that module's legend, and refreshes the 2 tables.
  • modulesPanel.js + the category/accordion half of viewportStats.js — delete the activeCategory ⇄ activeModule mutual-exclusivity glue; the dial replaces it.
  • liveBar.js — reduce to 2 tabs; keep Columns / Export only on the tabs that remain.
  • Keep untouched: plotsLayer.js, dataTable.js, taxonomy.js, state.js, Farm Analysis.

Risk: low. Reuses: the whole map layer. Test: switching the dial keeps map + legend + both tables in sync; no orphaned surfaces.

AModule hub medium · milestone 2

Nav lists the six modules by name. Home is a scorecard; each module owns a page built from one reusable template.

Changes

  • New js/dashboard/router.js — hash routes (buildless, no deps): #/overview (default), #/m/<key>, #/farm/<id>. Swaps the main content; the single Leaflet map is created once and re-parented / recoloured per route so the 4.4 MB geometry never reloads.
  • js/config.js — nav groups become Overview + the 6 modules + Farm Analysis. sidebar.js already renders this; just mark active by route.
  • New Home render (js/pages/overview.js) — six F3 cards from F2 rollups. No map.
  • New js/dashboard/modulePage.js — the reusable template: KPI strip (F1) + map (F2, one colouring) + legend + attention list (F4). Irrigation/Yield/Water reuse band colouring; Palms/Crop/Structures use taxonomy colouring. Build once, six routes.
  • index.html — strip the left stack + bottom tabs; add a #route-view mount and a persistent map container. Retire modulesPanel.js.
  • Reuses: plotsLayer colouring, modules.js bands, dataTable.js, Farm Analysis (as the #/farm route).

Risk: medium — routing + re-parenting one map across routes. Test: each route yields the right colouring + KPIs; scorecard numbers equal the rollups.

A2Map-led hub small delta on A · milestone 3

Everything in A, plus the Overview leads with the region map (all farm boundaries) and the six module cards become a launch strip beneath it.

Changes (delta over A)

  • Home render — mount the persistent map above the cards, framed. Default colour-by = land use, with a "Colour by" control (land use / crop / none) that recolours via plotsLayer.
  • Cards → the compact F3 mini variant, in a strip below the map. Clicking a tile routes to #/m/<key> (identical module page to A).
  • The Overview and module routes share the same map object — Overview shows it framed at the top, a module route shows it full-height.
  • Reuses: A's entire module page + F1–F4. The only net-new is one map instance on Home + the colour-by control + the card layout.
The one real risk

A2's Home map must draw all 25k boundaries at once — the binding performance constraint of this whole plan. Mitigations live in F2 (bulk add, cluster-below-threshold, canvas). If it still feels heavy in the room, A2 degrades to A by simply not mounting the Home map — zero wasted work, because the module pages are identical.

Test: Home renders map + 6 tiles; the colour-by switch recolours; a tile click routes to its module.

Sequencing & effort

The order Foundation → B → A → A2 is additive — nothing built early is discarded. B's viewSelector logic and the F1 registry feed A's module pages; A's module page and Home cards feed A2; A2 only adds the Home map.

If the GITEX clock is tight

Build F1 + F2, then jump straight to A2's Home map and the one module page that is the demo gate — Palms & Fruit Trees (31.6%, the module Mark flagged). The other five module pages reuse that same template afterward. You get the map-first landing and the critical module without waiting on the full six-page build.

Two things to keep honest throughout

  • The mock boundary. All new metrics (canopy health, tree count, tier, cultivated/fallow) go in js/mock/* and data/* so the whole thing stays swappable for a real API — exactly as the README's "mock boundary" already sets up.
  • Tests before each layout flip. Per CLAUDE.md: add regression tests for F1–F4 before ripping out the accordion/tab machinery, so the refactor can't silently drop a module or a metric.
